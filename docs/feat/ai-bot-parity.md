# AI bot parity and vision audit

**Date:** 2026-07-23  
**Audited checkout:** `b289793` (`chore/gitignore-sim-reports`)  
**History reviewed:** bot/sim work through PR #41, including off-checkout refs  
**Scope:** the shipped game engine and every consequential choice a human player can make or inspect. Read-only UI conveniences such as opening the Codex are not bot actions; the underlying rules and state they expose are in scope.

## Verdict

The engine has **execution parity, not decision parity**.

- **Execution is strong:** all **27 mutating/turn move types** are represented by the typed `LegalMove` union, enumerated through the engine's status predicates, and dispatched to the real mutators. Automatic income, season, event, arrival, unrest, and victory processing runs through the same turn machine for humans and bots. See [`legalMoves.ts`](../../src/game/legalMoves.ts#L90) and [`turn.ts`](../../src/game/turn.ts#L81).
- **Vision is partial:** policies receive the entire raw `HegemonyState`, but the strongest economic evaluator only turns a small part of it into judgment: own resources, six copies of the _current_ income vector, own settlements/pops/slots, current victory-card count, happiness, and Influence. Raw availability is not useful perception. See [`policies.ts`](../../src/sim/policies.ts#L170).
- **The new `master` policy composes the existing specialists:** it runs the political Assembly handler and otherwise uses beam search over Smart economics + political standing + PR #41's one-step frontier signal. This closes the previous sibling-policy fragmentation, but it is not yet a parity-grade reference bot: cross-turn saving, general rival replies, multi-hop route search, and chance expected value remain unbuilt.
- **Default simulations are not representative games:** `auto`/`batch` default to `random`, the sim defaults to the Classic board while the live game defaults to Shuffled, normal sim setup is chosen uniformly at random rather than by the selected policy, and the default 40-turn batch barely reaches the Year-2 Assembly. See [`cli.ts`](../../src/sim/cli.ts#L173), [`cli.ts`](../../src/sim/cli.ts#L536), and [`setup.ts`](../../src/sim/setup.ts#L66).
- **The largest remaining blind spots are strategic, not mechanical:** opening placement, multi-hop/directional expansion, racing rivals to scarce tiles, saving across turns, victory-card margins and denial, event-coupon follow-through, chance expected value, pivotal Assembly voting, and Stratokles play.
- **The simulator can over-see:** the policy contract exposes future shuffled event and politician decks in raw state. Current policies generally refrain from reading them, and search refuses to branch through seeded random outcomes, but there is no observation boundary that prevents a future policy from cheating. See [`state.ts`](../../src/game/state.ts#L73).

### Policy capability matrix

All policies receive the same raw state and legal moves. The difference is which facts their decision function actually values and how far it searches. A check means deliberate reasoning, not merely that the action can be selected legally.

### Status key

| Status          | Meaning                                                                                                    |
| --------------- | ---------------------------------------------------------------------------------------------------------- |
| **Covered**     | The action is available and the best current policy has a reasonable value signal for it.                  |
| **Partial**     | Legal access exists, but perception, valuation, planning, or telemetry is materially incomplete.           |
| **Access only** | The bot can execute the action but normally cannot tell when or why it is good.                            |
| **Missing**     | No credible bot behavior or parity guard exists.                                                           |
| **Off-branch**  | A useful experiment exists in git history, but it is not in this checkout and does not close the full gap. |

## Recommended implementation order

1. **Create the parity contract before another heuristic.** Add a feature/capability manifest and `movesByType` telemetry. Make CI require action, observation, behavior fixture, and instrumentation for every consequential feature.
2. **Introduce a non-cheating observation/query layer.** Redact ordered decks/private cards; expose authoritative income forecasts, victory margins, public chance distributions, map reach/threat queries, and Assembly ballot state.
3. **Evolve `master` into the parity-grade reference policy.** The first composition now combines beam, politics, and the measured frontier signal; next add temporal planning, the general opponent model, multi-hop routes, and chance EV. Keep the specialist policies as controlled probes.
4. **Make setup strategic and profiles honest.** Route random opening through seat policies for reference runs; add named `regression` and `live-balance` profiles; require/strongly signal policy, board, opening, and late-system exposure in reports.
5. **Add cross-turn planning.** Permit the planner to deliberately end a turn, receive a non-peeking own-income forecast, reserve resources, and pursue a serialized objective such as city upgrade or victory-card swing.
6. **Add opponent/victory/map vision.** Exact card margins, rival threat summaries, contested-tile arrival races, 2–4-step expansion corridors, and future city anchors should share one competitive evaluation.
7. **Handle chance by expectation.** Riot insurance, ventures, future public decks, and event choices should branch over public distributions or expected utility—not seeded outcomes and not fixed rules of thumb.
8. **Deepen Assembly decisions.** Model passage/pivotality, minimum bribe, veto timing, field-wide public-good subsidy, exact Voice completion, delayed effects, and intentional Stratokles/coup play.
9. **Tune only after behavior passes.** Re-run headline balance campaigns on Shuffled boards with strategic openings, parity-grade `master`, all systems exposed, seat rotation, and behavior/telemetry coverage attached to the report.

## Definition of parity for every future feature

A feature is bot-parity complete only when every applicable cell below is green:

| Layer         | Required proof                                                                                                                               |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **Act**       | Every human mutation has a typed legal move or is explicitly automatic; enumeration and application use the authoritative engine.            |
| **Observe**   | The bot gets every public fact and derived query a competent human could use, with private/future information redacted.                      |
| **Value**     | The `master` evaluator represents immediate payoff, delayed payoff, risk, opportunity cost, victory impact, and rival impact where relevant. |
| **Plan**      | A targeted scenario proves the bot can sequence prerequisites or wait/save when the feature requires it.                                     |
| **Execute**   | Deterministic games exercise the action without deadlock, illegal moves, state mutation during choice, or seeded-outcome peeking.            |
| **Measure**   | Reports expose action frequency, choice breakdown, context, realized outcome, and feature-specific success—not merely that a verb fired.     |
| **Integrate** | The capability is present in `master`, the single policy used for whole-game balance, not only in a specialist A/B bot.                      |
| **Regress**   | CI includes clearly-use, clearly-avoid, edge-case, hidden-information, and replay fixtures.                                                  |

Until all eight layers are present, a simulation result should be labeled as measuring **the game under a named limited policy**, not the game in general.
