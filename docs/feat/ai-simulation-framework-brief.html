AI Simulation Framework

Current implementation, comparison with established approaches, and recommended next steps.

1. Current state of the simulation

The simulation uses the real game rules and legal moves. The current master bot uses a hand-written position score and searches up to four actions within its current turn. It does not plan across full rounds, model rival responses, or calculate expected values for random outcomes.

Rules and execution

  • The simulator enumerates legal actions from the same rules used by the player-facing game.
  • Hypothetical and real actions use the same move application functions.
  • Games are seed-driven and reproducible.
  • Different bot policies can be assigned to individual seats.
  • Random setup currently chooses uniformly from legal placements instead of using the selected bot policy.

Master policy

master combines the current specialist logic:

  • Economic evaluation: resources, income, populations, settlements, building slots, happiness, and victory cards.
  • Within-turn search: beam width 3, depth 4.
  • Political evaluation: Assembly resolutions, patronage, political leads, and Stratokles.
  • Spatial evaluation: the total printed yield on tiles where the player could found its next colony.

Benefit calculation

For deterministic actions:

estimated benefit = score(state after action) - score(current state)

The economic evaluator first projects each resource:

projected resource = current stockpile + 6 × current income

This is not a six-turn simulation. It assumes that the current income repeats six times without changes from later construction, opponents, laws, or events.

Political calculation

The bot enacts a proposed resolution on a cloned state, recalculates every player's position, and uses: 

resolution value = my score change - largest rival score change

This captures immediate and recurring effects visible to the normal evaluator. It does not reliably capture later discounts, future free actions, vote pivotality, multi-round use, or spatial effects. Master currently uses the Political evaluator, not the full Master evaluator, while inside the Assembly.

Current limitations

  • No cross-turn saving or planning.
  • No general opponent-response model.
  • No multi-hop colony route planning or opponent map races.
  • No expected-value search for riots, ventures, events, or hidden draws.
  • No observation boundary separating public information from hidden engine state.
  • Recurring influence income is not properly included in the projected material score.
  • Evaluation weights are heuristic points, not calibrated win probabilities.
  • Behavioral tests prove determinism and legality, but not full strategic awareness for every feature.

Current use: deterministic rules testing, basic balance simulations, policy comparison, and a baseline bot. It is not yet sufficient as the final player-facing CPU.

2. Comparison with other frameworks

The current system is closest to utility AI combined with shallow beam search. For a multiplayer game with random outcomes and possible hidden information, the most relevant next framework is Monte Carlo Tree Search with an information-set or belief-state layer.

Framework How it decides Strengths Limitations for Hegemony
Current Master Hand-written utility score plus width-3, depth-4 beam search. Fast, deterministic, explainable, and integrated with the real rules. No full rounds, rival replies, chance sampling, or hidden-information boundary.
Rule or utility AI Scores available actions with authored rules and weights. Cheap, controllable, easy to give personalities and difficulty levels. Misses long combinations unless they are explicitly encoded.
Minimax / alpha-beta Searches alternating moves under adversarial assumptions. Effective in deterministic, two-player, zero-sum games. Poor fit for four players, chance, politics, large action sequences, and hidden information.
Monte Carlo Tree Search Runs many sampled future games and uses their outcomes to focus search. Handles large action spaces, multiple players, and random outcomes. Needs a fast simulator, action pruning, opponent policies, and a time budget.
Information Set MCTS Searches over plausible states consistent with what the player knows. Prevents cheating when cards, decks, or information are hidden. Adds belief-state and sampling complexity.
AlphaZero-style self-play Learns policy and value models from self-play and uses them to guide search. Can learn long-term patterns not captured by hand-written scoring. High implementation and training cost; standard form assumes a simpler two-player, deterministic setting.

Catan comparison

A published Catan agent used Monte Carlo Tree Search with 1,000 to 10,000 simulated games per real move. Its simulations included future player actions and random outcomes. At 10,000 simulations it won 49% of four-player games against three JSettlers heuristic agents. An equal seat would be expected to win about 25%.

The study also found that poorly chosen heuristic biases reduced playing strength. This is relevant to the current Master weights: adding more hand-written values is not automatically an improvement. Weights must be tested through controlled tournaments.

Recommended framework choice

Do not replace the current evaluator. Use it as the default action policy and leaf evaluator inside a bounded stochastic MCTS system. Add an information-set layer if the CPU must reason about hidden cards or decks. Consider learned policy or value models only after this search and measurement infrastructure is stable.

3. Best steps moving forward

  1. Create a bot observation boundary.

    Define exactly what each player can see. Policies should receive public state, their own private information, and explicit beliefs—not the ordered future decks or other hidden engine state.

  2. Return a structured score breakdown.

    Replace opaque scalar-only output with economy, victory, happiness, spatial, political, risk, and opponent components. Record these reason codes in simulation telemetry. Correct recurring influence valuation and make Master use its full evaluator in the Assembly.

  3. Add expected-value handling for random actions.

    Enumerate small outcome tables and sample large ones. Evaluate expected payoff, variance, and downside without reading the actual next seeded result.

  4. Extend planning across complete rounds.

    Include end turn, income, opponent turns, Assembly responses, map races, and delayed purchases. Use the existing policies as initial opponent models.

  5. Implement bounded MCTS around Master.

    Master supplies action ordering, rollout behavior, and unfinished-state evaluation. A simulation or time budget controls difficulty and runtime.

  6. Add capability tests for every game feature.

    For each mechanic, create clear use, avoid, and edge-case scenarios. Assert that Master perceives the relevant effect and returns the expected reason codes.

  7. Measure with paired tournaments.

    Compare agents on identical seeds with rotated seats and shuffled boards. Track win rate, rank, feature usage, forced turns, runtime, and confidence intervals. Tune weights only after capability tests pass.

  8. Add CPU difficulty and personality after search works.

    Vary simulation budget, opponent-model quality, risk preference, and near-best move selection. Keep specialist policies for testing; use the cumulative agent as the player-facing CPU base.

Immediate order: observation boundary → score breakdown and fixes → chance expected value → full-round opponent simulation → bounded MCTS → capability tests and tournaments → CPU difficulty and personality.
