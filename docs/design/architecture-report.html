Hegemony · Engineering Review · July 2026

Frontend Architecture Report

Component health, the engine/UI boundary, and the road to an online game — audited on feat/phase1-currencies @ 8d609f8, ~4,950 LOC of UI across 24 component files, 4,502 lines of CSS.

Companion to the Hegemony Brandbook v0.1 — the refactor order below is sequenced so the visual reskin lands on shared components instead of nine copies of everything.

IThe Verdict

Your instincts were right on all four counts, but the news is better than the hunch. The UI code is messier than it should be — nine hand-copied modal scaffolds, a 17-prop component, four implementations of the same resource chip — yet the architecture underneath is already the one an online game needs. The engine is pure, serializable, deterministic, and drivable through a data-only move protocol that could travel over a websocket today. The gap between here and multiplayer is organizational, not structural.

One number that summarizes the audit: exactly one file in src/game is impure — controller.ts, the React hook layer, which merely lives in the wrong directory. Evicting it is the only semantic change the whole frontend/backend separation requires. Everything else is mechanical renames.

IIDependency Reality

Measured from every import statement in src/ — no back-edges exist anywhere. The layering you'd design on a whiteboard is already the layering on disk, with one tenant on the wrong floor:

  • src/sim is the proof of portability. It plays entire games with enumerateLegalMoves → applyMove → endTurn and node builtins alone. It is, functionally, the prototype of your future authoritative server loop.
  • Assets are the only other fragile coupling: three files reference art via depth-relative new URL("../../../assets/…") — a pattern that already broke once (there's an apology comment in board/events.ts). Move to public/ + absolute paths before any restructure.
  • All 18 test files are node-environment engine/sim tests. Extraction won't disturb them beyond an include-glob.

IVMultiplayer Readiness

Scored against what a server-authoritative online game with optimistic client prediction actually requires:

RequirementStatusDetail
Serializable statePassHegemonyState is pure JSON — no functions, classes, Maps, Dates. Survives JSON.parse(JSON.stringify(G)) losslessly. RNG state is a number on the state object.
Deterministic replayPassmulberry32 with explicit state threading; the only Math.random() is createSeed() — the entropy source a server would own. Same seed + same moves = same state.
Wire-ready move protocolExists, unused by UILegalMove is a plain discriminated union; applyMove(G, playerID, move) is the uniform dispatcher. The sim and bots use it. The React UI bypasses it via direct mutator closures — converging the UI on applyMove is the single biggest step toward networking.
Authoritative loopProvensim/runner.ts already runs validate → apply → telemetry headlessly. The server is a transliteration, not an invention.
Optimistic rollback shapeAccidental passThe controller's Immer commit-on-ok pattern (keep previous on rejection) is exactly optimistic-apply-with-rollback, waiting for a server ack to slot in.
Hidden informationMissingAll four draw piles sit ordered inside the one state object every client would receive — full deck order leaks. Needs a per-player playerView redaction (or server-held decks, clients see counts + discards).
Move authorizationMissingMoves bake in G.currentPlayer; any seat can drive any seat (hot-seat feature, online bug). applyMove already takes an explicit playerID, so only the controller side needs auth.

VRepo Structure — the mprocs Question

Three end-states were compared: (A) single package with enforced boundaries, (B) npm-workspaces monorepo (engine / client / server / sim), (C) two separate repos with a published engine.

Recommendation: B — the workspaces monorepo — reached in four stages, where Stage 0 is all of A's value. Not C: a published-package cycle inserts commit → publish → bump → sync into the sim-driven balance loop you run daily (npm run sim against live engine source), and that is the most expensive thing you could do to yourself mid-balancing. Separate repos are for separate teams; you are one person with one lockfile.

Stage 0
hours · do now

Purity lock-in. Move src/game/controller.ts → src/ui/controller.ts (18 mechanical import updates — the only semantic change in the entire migration). Add eslint boundary rules (game may not import react/ui/components) and a tsconfig.engine.json with no DOM lib over game+sim as a CI gate — compiler-guaranteed purity, catching what lint can't.

Stage 1
half a day

De-fragilize. Path aliases (@game/*, @ui/*) replace the three-deep relative imports; assets/ moves to public/ with absolute URLs (3 files). After this, moving files stops breaking things — which matters, because the reskin will move files.

Stage 2
~1 day

Workspaces. packages/engine (pure TS, source-level exports, no build step — moduleResolution: "Bundler" resolves .ts directly for vite/tsx/vitest), packages/sim, packages/client. The sim doubles as the extraction acceptance test: npm run sim must reproduce the saved Q13b baselines byte-identical. Engine's package.json carries no react and no immer — purity enforced by the package manager itself.

Stage 3
when multiplayer starts

Server + mprocs. packages/server: websocket transport around the sim-runner loop — produce(G, draft => applyMove(draft, player, move)), validate against G.currentPlayer, redact draw piles per client, broadcast. Your GameMoves vocabulary is already the wire protocol, verbatim.

# mprocs.yaml
procs:
  client:  { shell: "npm -w @hegemony/client run dev" }
  server:  { shell: "npm -w @hegemony/server run dev" }
  check:   { shell: "npm run -ws check -- --watch" }
  sim:     { shell: "npm -w @hegemony/sim run sim -- --watch" }

Every stage leaves dev, test:run, sim, and build green — you can stop at any rung if priorities shift, and nothing is thrown away by the next rung.

VIHow This Meets the Brandbook

The two workstreams interlock rather than queue. The efficient sequence:

  • First R1–R3 + Stage 0–1 (a weekend of work): one modal shell, one chip, one verb component, purity gates, stable paths. Now the surface area of "what must be themed" has collapsed from ~30 call sites to ~6 components.
  • Then the brandbook's token migration (tokens to :root, alpha-literals derived from tokens, the 8-role type scale) lands once, on the consolidated components — instead of being re-applied nine times.
  • Then the asset regeneration batch (one icon language, Phase-1 gaps filled) slots into the existing mask-tint pipeline, which survives untouched.
  • Stage 2–3 wait until the reskin has shipped and multiplayer work actually begins — the monorepo is a day of renames whenever you want it.
